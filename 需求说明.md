自服务接口账号数量限制需求设计

# 需求背景
1. 自服务系统账号结构：一个企业可创建多个登录用户；每个登录用户下可绑定多个接口账号（API账号）。接口账号用于发送短信，登录用户在页面可查看其名下接口账号的发送数据。
2. 客户“石基时代”为酒店短信发送代理商：会为每家酒店创建一个 API 账号，与我司对接发送短信。
3. 客户管控诉求：希望按“API账号”维度限制可发送量，用于管控各酒店的短信发送；不希望改变企业整体后付费结算模式。
4. 结算边界：本需求仅是业务侧发送管控动作，不作为计费/结算依据；企业与我司仍按后付费结算。
5. 自助化要求：上述配置与操作需要开放给客户，在自服务平台自主完成。

# 需求目标（业务定义）
- 对单个 API 账号的短信发送进行“可发送量管控”（发送闸门）。
- 管控仅用于限制发送行为，不影响企业与我司的实际结算方式（仍为后付费结算）。

# 管控对象与口径
- 管控对象：API 账号。
- 计数口径：按长短信拆分后的“计费条数/段数”口径计数（与现有长短信拆分规则一致）。
- 计数时机：当发送请求被我司“成功受理”后计入已用量并扣减可用量。

# 管控规则
- 支持开启/关闭管控：
  - 关闭时：该 API 账号发送不受本规则限制。
  - 开启时：该 API 账号存在“初始值（上限）”与“已用量”；剩余可发 = 初始值 - 已用量（不足按 0 计）。
- 支持两种模式（可选）：
  - 一次性模式：初始值表示该 API 账号的一次性总可发量。
  - 周期模式：初始值表示每周期可发量；支持按日/按月两种周期。
- 初始值修改规则：
  - 允许在周期中修改初始值；修改后立即生效，并按“剩余可发 = 新初始值 - 已用量（不足按 0 计）”重新计算。

# 接口返回约定
- 当余量不足/达到上限导致拒绝受理时，返回错误码：GB:0050。
- 错误码解释：自服务接口账号发送上限。

# 自服务页面需求（API账号管理）
- 列表展示：管控状态、剩余可发(段)、已用(段)、管控模式（一次性/周期-按日/按月）。
- 操作能力：
  - 发送管控设置：开启/关闭、模式选择、周期类型（按日/按月）、初始值设置与修改。
